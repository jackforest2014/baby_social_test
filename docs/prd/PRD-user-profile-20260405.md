## 2. 产品概述

### 2.1 背景

BabySocial 是一款面向全行业用户的社交平台应用。用户注册与登录功能（PRD-USER-AUTH-001）已完成定义，个人主页展示功能是第二个核心功能模块。

个人主页承担双重职责：一方面是用户对外展示自我形象的窗口（头像、昵称、简介、专长等基础信息）；另一方面承载了一个差异化核心功能——**个人短链 + AI 助理聊天**。任何人（无需登录）都可以通过形如 `babysocial.com/will` 的个性化短链访问该用户的 AI 助理聊天界面，AI 助理会依据用户上传的 profile 文件来代替用户接待访客。

这一设计让用户的个人主页从静态名片升级为可交互的智能代理门户。

### 2.2 目标

- 为用户提供清晰的个人信息展示界面，支持头像、昵称、简介等基础资料的填写与展示。
- 通过注册引导流程 + 完善度评分，鼓励用户填写更完整的个人资料，提升平台内容质量。
- 为每位用户提供唯一的个人短链，支持以公开聊天界面的形式对外展示 AI 助理。
- 支持用户上传 profile 文件，后端自动生成展示摘要并将文件作为 AI 助理知识库。
- 在架构层面预留关注、私信、作品、动态等后续社交功能的扩展点。

### 2.3 目标用户

- **平台注册用户（本人）**：需要管理和展示个人资料、上传 profile、配置 AI 助理的用户。
- **已登录的其他平台用户**：浏览他人主页，了解对方基本信息。
- **匿名访客（无需登录）**：通过个人短链访问 AI 助理聊天界面，与对方的 AI 助理进行交互。

## 3. 功能需求

### 3.1 个人主页展示

#### 3.1.1 访问规则

- 个人主页路由：`/profile/{user_id}` 或 `/profile/{slug}`（slug 即个人短链标识符）。
- **主页需要登录才能访问**。未登录用户访问主页 URL 时，重定向至登录页，登录完成后跳回原 URL。
- 已登录用户可以访问任意其他用户的主页（只读）。
- 用户本人访问自己主页时，显示"编辑资料"入口。

#### 3.1.2 展示字段

以下所有字段均为可选，用户未填写时页面对应区域不显示或显示占位提示：

| 字段分类 | 字段名称   | 说明                                                                                       |
| -------- | ---------- | ------------------------------------------------------------------------------------------ |
| 基础信息 | 头像       | 用户上传的图片，未上传时显示系统默认头像（以邮箱首字母生成）                              |
| 基础信息 | 昵称       | 用户设置的显示名称，未设置时默认显示邮箱前缀                                              |
| 基础信息 | 个人简介   | 不超过 200 字的自由文本                                                                    |
| 基础信息 | 用户专属短链 | 系统注册时自动生成，显示为 URL（`babysocial.com/profile/ai/{short_id}`），旁边提供"一键复制"按钮，所有访客可见 |
| 职业背景 | 专长       | 用户擅长的技能或领域标签（多选），如"前端开发"、"产品设计"                                |
| 职业背景 | 职业       | 自由文本，如"独立开发者"、"UI 设计师"                                                     |
| 个人偏好 | 兴趣       | 用户感兴趣的话题标签（多选），如"健身"、"阅读"                                            |
| 个人偏好 | 喜好       | 自由文本或多选标签，描述生活偏好                                                          |
| 个人信息 | 性别       | 选项：男 / 女 / 不公开，默认不公开。选择"不公开"时对任何人都不展示                      |
| 个人信息 | 生日       | 日期选择，主页只展示年龄（"X 岁"），不直接展示生日日期                                   |
| 联系方式 | 联系方式   | 自由文本，用户自行填写（如微信号、邮箱、个人网站等），不做格式强制约束                    |
| AI 助理  | profile 摘要 | 基于用户上传的 profile 文件由后端生成的摘要内容，仅展示摘要，不展示原始文件             |
| AI 助理  | 作品       | [待后续迭代] 预留区域，MVP 阶段不展示                                                     |
| 社交数据 | 动态       | [待后续迭代] 预留区域，MVP 阶段不展示                                                     |

#### 3.1.3 资料完善度分数展示

- 个人主页右上角（或资料区顶部）展示"资料完善度"进度条和分数（如"完善度 65%"）。
- 分数仅对用户本人可见，他人浏览时不显示。
- 分数旁显示"继续完善"按钮，点击跳转至资料编辑页面并高亮未填写字段。

### 3.2 个人资料编辑

#### 3.2.1 编辑权限

- 仅用户本人可以编辑自己的资料。
- 他人主页不展示编辑入口，且后端 API 对编辑接口进行身份校验，非本人请求返回 `403 Forbidden`。

#### 3.2.2 编辑入口

- 用户本人访问自己的主页时，页面顶部右侧显示"编辑资料"按钮。
- 注册引导流程（见 3.3 节）也会引导用户进入资料编辑页面。

#### 3.2.3 编辑页面字段

编辑页面包含以下可编辑字段，均为可选项：

| 字段       | 控件类型           | 限制说明                                    |
| ---------- | ------------------ | ------------------------------------------- |
| 头像       | 图片上传           | 格式：JPG / PNG / WebP，大小 <= 512KB，自动裁剪为 1:1 |
| 昵称       | 文本输入框         | 1-50 字符，不允许纯空格                     |
| 个人简介   | 多行文本框         | 最多 200 字，超出时前端实时提示剩余字数     |
| 专长       | 多选标签 + 自定义  | 最多选 10 个标签，每个标签最长 30 字符      |
| 职业       | 文本输入框         | 最多 50 字符                                |
| 兴趣       | 多选标签 + 自定义  | 最多选 10 个标签，每个标签最长 30 字符      |
| 喜好       | 多选标签 + 自定义  | 最多选 10 个标签，每个标签最长 30 字符      |
| 性别       | 单选（男/女/不公开）| 默认不公开                                 |
| 生日       | 日期选择器         | 年龄须在 6-120 岁范围内                     |
| 联系方式   | 文本输入框         | 最多 200 字符，自由格式                     |

#### 3.2.4 保存逻辑

- 编辑页面提供"保存"按钮，点击后提交全量更新（或仅修改字段的增量更新，由后端实现决定）。
- 保存成功后，显示成功提示并返回主页视图。
- 保存失败时，显示具体错误原因，不清空用户已填写内容。

### 3.3 注册后引导填写资料

#### 3.3.1 触发时机

- 用户完成注册并自动登录后（见 PRD-USER-AUTH-001 步骤 6），系统立即跳转至"引导填写资料"页面，而非直接进入平台首页。
- 引导页仅在用户首次注册后触发一次，后续登录不再触发。

#### 3.3.2 引导页面设计

- 引导页采用分步表单形式（Step 1 / Step 2 / Step 3），每步聚焦 2-3 个字段，降低填写压力。
- 每一步右上角显示"跳过此步"按钮，用户可随时跳过当前步骤。
- 最后一步（或每步跳过时）显示"完成，进入主页"按钮，用户可随时退出引导流程。
- 退出引导后跳转至用户的个人主页。

**分步示例**（步骤划分可在交互设计阶段调整）：

| 步骤   | 引导填写的字段     |
| ------ | ------------------ |
| Step 1 | 头像、昵称、个人简介 |
| Step 2 | 专长、职业、兴趣、喜好 |
| Step 3 | 性别、生日、联系方式、个人短链 |

#### 3.3.3 进度提示

- 引导页顶部显示步骤进度（如"第 2 步，共 3 步"）和当前资料完善度评分的实时变化，激励用户继续填写。

---

### 3.4 资料完善度评分机制

#### 3.4.1 计算规则

完善度分数满分 100 分，按字段是否填写给予对应权重分值，累计得分后计算百分比：

| 字段       | 权重（分） | 判定填写完成的条件                              |
| ---------- | ---------- | ----------------------------------------------- |
| 头像       | 15         | 已上传自定义头像（非默认系统头像）              |
| 昵称       | 10         | 已填写且非空白                                  |
| 个人简介   | 10         | 已填写且字符数 >= 10                            |
| 专长       | 10         | 至少选择 1 个标签                               |
| 职业       | 10         | 已填写且非空白                                  |
| 兴趣       | 5          | 至少选择 1 个标签                               |
| 喜好       | 5          | 至少选择 1 个标签                               |
| 性别       | 5          | 选择了"男"或"女"（选择"不公开"不计分）         |
| 生日       | 5          | 已填写生日日期                                  |
| 联系方式   | 5          | 已填写且非空白                                  |
| 用户专属短链 | 10       | 系统注册时自动生成，此项自动满足（注册即得分）  |
| Profile 文件 | 10       | 已上传至少一份 profile 文件，且摘要生成完成     |

> 说明：以上权重合计 100 分。各字段权重可在产品迭代中根据数据调整。

#### 3.4.2 展示规则

- 分数计算在客户端实时完成（根据已保存的字段状态），无需额外 API 请求。
- 展示格式：进度条 + 百分比数字，颜色随分数段变化（0-40% 红色、41-70% 橙色、71-100% 绿色）。
- 完善度 100% 时，显示"资料已完善"徽章，进度条颜色变为绿色并有完成动画效果。
- 分数仅对本人可见，不对其他用户展示。

### 3.5 个人短链功能

#### 3.5.1 短链规则

- 用户专属短链，**由系统在用户注册成功时自动生成**，用户不可修改。
- 格式：`babysocial.com/profile/ai/{short_id}`，其中 `short_id` 为系统生成的唯一标识符。
- **字符规范**：仅包含英文字母（a-z）和数字（0-9），不含特殊字符。
- **长度限制**：不超过 16 个字符。
- **唯一性**：系统保证全平台唯一，生成时需排除与系统路由关键词的冲突（如 `api`、`admin`、`login` 等）。
- **不可修改**：短链一经生成，终身绑定该用户，不支持修改或自定义。
- **可见性**：短链显示在用户主页上，**所有访客和其他用户均可见**，旁边提供"一键复制"按钮。

#### 3.5.2 短链访问逻辑

- 用户访问 `babysocial.com/profile/ai/{short_id}` 时：
  1. 服务端查找该 short_id 对应的用户。
  2. 若找到，直接渲染 AI 助理聊天界面（无需登录，公开访问，见 3.6 节）。
  3. 若未找到，返回 404 页面，提示"此主页不存在或已下线"。
- `babysocial.com/profile/ai/{short_id}` 路由优先级低于系统路由（如 `/api`、`/login` 等）。

### 3.6 AI 助理聊天界面

#### 3.6.1 访问方式

- **通过用户专属短链访问**：`babysocial.com/profile/ai/{short_id}`（每位用户注册时自动生成）。
- **访问权限**：完全公开，无需登录，任何人均可访问。

#### 3.6.2 界面功能

- 聊天界面为简洁的单页对话界面，包含：
  - 顶部：主人的头像、昵称（或默认"XXX 的助手"），以及一句话简介（取自 profile 文件或主人填写的简介）。
  - 中部：聊天消息列表，**AI 助理在访客进入页面时率先发送欢迎语**（无需访客先发消息），欢迎语基于主人的 profile 文件生成。
  - 底部：文本输入框 + 发送按钮，支持回车键发送。
- AI 助理依据主人上传的 profile 文件（知识库）来回应访客的提问。
- 若主人未上传 profile 文件，AI 助理以通用友好话术接待，告知访客"主人尚未配置详细介绍，欢迎留言"。

#### 3.6.3 AI 助理行为规范

- AI 助理只回答 profile 文件中明确写出的内容；profile 文件未涵盖的问题，AI 助理统一回应"这个问题由主人本人来回答更合适"。
- AI 助理不能替主人承诺任何事项（如"我会接受你的合作"），只能表达意向性信息。
- **对话内容需要持久化存储**，页面刷新或重新访问时，历史对话可恢复。对话记录存储于 `ai_chat_sessions` 和 `ai_chat_messages` 表中（见第 6 节）。

#### 3.6.4 AI 模型与调用

- MVP 阶段默认使用 **qwen-plus**（通义千问）作为 AI 助理的大模型。
- 系统支持**通过配置替换为其他模型**，无需修改代码。当前备选的免费/低成本模型包括：
  - **Gemini 2.0 Flash**（`gemini-2.0-flash`）：Google 免费模型，每分钟 15 次请求、每天 1500 次，速度快，适合对话场景，推荐作为备选。
  - **Gemini 2.0 Flash-Lite**（`gemini-2.0-flash-lite`）：更轻量，延迟更低，免费额度与 Flash 类似。
  - **Gemini 1.5 Flash**（`gemini-1.5-flash`）：128K 上下文，免费层每分钟 15 次、每天 1500 次。
  - **Gemini 1.5 Pro**（`gemini-1.5-pro`）：100 万上下文窗口，免费层每分钟 2 次、每天 50 次（额度较低，不推荐高频使用）。
  > 注：以上 Gemini 免费额度信息基于 2025 年 5 月数据，部署前需到 [ai.google.dev](https://ai.google.dev) 确认最新政策。
- 模型切换通过 `AIProvider` 接口抽象 + 配置文件实现（详见第 5.3 节）。
- 后端对每次对话请求进行频率限制，防止滥用：同一 IP 每分钟最多 20 次请求，超出时返回 `429 Too Many Requests` 并提示用户稍后再试。
- 前端不直接暴露 API Key，所有 LLM 调用通过后端代理。

### 3.7 Profile 文件上传与总结生成

#### 3.7.1 文件格式与规范

- 参考格式：项目中 `docs/requirements/visitor_intent/owner_profile.md` 所示的结构化 Markdown 文件。
- **支持上传的文件格式**：`.pdf`（PDF）、`.md`（Markdown）。页面上传区域需明确提示用户："当前仅支持 PDF 和 Markdown 格式，更多格式将在后续版本支持"。
- 文件大小限制：不超过 1MB。
- 每位用户 MVP 阶段仅支持一份有效 profile 文件（上传新文件后覆盖旧文件）。

#### 3.7.2 上传入口

- 个人主页本人视图和资料编辑页均提供"上传 Profile 文件"按钮。
- 点击后打开文件选择器，用户选择文件后立即上传，上传过程显示进度条。

#### 3.7.3 后端处理流程

```
用户上传文件
    ↓
后端接收文件，校验格式和大小
    ↓
文件存储至 PostgreSQL 数据库（后续可通过抽象层切换为 OSS/S3 等，详见技术方案）
    ↓
异步触发摘要生成任务（调用 LLM）
    ↓
LLM 生成结构化摘要（不超过 200 字，列表形式）
    ↓
摘要存储至数据库（user_profile_files 表）
    ↓
同时将文件内容作为该用户 AI 助理的知识库（以 <profile>...</profile> 注入 prompt）
    ↓
主页 profile 摘要区域显示生成结果
```

#### 3.7.4 展示规则

- 主页上展示的是 LLM 生成的**摘要内容，以列表（要点列表）形式展示**，不展示原始文件内容，也不提供原始文件下载链接（对访客不可见）。
- 摘要生成期间（异步处理中），主页该区域显示"Profile 摘要生成中，请稍候…"的占位提示。
- 摘要生成失败时，显示"摘要生成失败，请重新上传"，并提供重试按钮。
- 用户上传新文件后，旧摘要替换为新摘要（异步更新期间展示旧摘要 + "更新中"提示）。

#### 3.7.5 AI 知识库更新

- profile 文件上传完成后，该文件的内容作为该用户 AI 助理的知识库。
- **知识库接入方式**：MVP 阶段将 profile 文件内容直接作为 prompt 的一部分注入，格式为 `<profile>...</profile>` 标签包裹。具体 prompt 设计将在架构设计/技术方案阶段讨论。
- AI 助理在聊天时，优先依据 profile 文件内容回答；文件内容未涵盖的问题不作延伸回答。
- 知识库更新为实时生效：用户上传新 profile 文件后，新文件内容立即替换旧内容用于后续对话。

## 4. 非功能需求

### 4.1 安全性

- 所有资料编辑 API 必须通过 JWT 认证，并验证操作用户与目标资源的归属关系，非本人操作返回 `403 Forbidden`。
- 头像和 profile 文件上传需校验文件 MIME 类型（不仅依赖扩展名），防止伪造文件类型上传恶意内容。
- 上传文件需在服务端校验文件类型和大小。
- AI 助理聊天界面虽然公开访问，但后端对输入内容长度做限制（单次输入最多 500 字符），防止 prompt injection 或超长请求攻击。**预留内容安全审核接口**（`ContentModerator` 接口），MVP 阶段不实现审核逻辑，后续可接入第三方安全服务。
- 用户输入的昵称、简介等所有文本字段，后端必须进行 XSS 过滤和 SQL 参数化处理。
- AI 调用的 API Key 存储在服务端环境变量中，禁止硬编码，禁止在前端暴露。
- **头像和 profile 文件当前存储在 PostgreSQL 中**，技术方案中将对存储层进行抽象封装（`FileStorage` 接口），方便后续切换为 OSS/S3 等对象存储方案。

### 4.2 性能

- 个人主页加载时间（首次访问，含 CDN）P99 < 1s。
- 资料编辑保存接口响应时间 P99 < 500ms。
- AI 助理聊天首条回复延迟 P50 < 3s，P99 < 8s（受 LLM 接口响应时间影响）。
- Profile 摘要生成为异步任务，目标在上传后 60 秒内完成生成，用户无需等待。
- 头像图片上传后，后端自动压缩并输出 WebP 格式，控制文件大小，提升加载速度。

### 4.3 可用性

- AI 助理聊天界面可用性 >= 99.5%（因依赖第三方 LLM 服务，略低于核心功能）。
- LLM 服务不可用时，AI 助理聊天界面给出明确提示（"助手暂时不在线，请稍后再试"），不静默失败。
- Profile 摘要生成失败时，主页给出明确失败提示，支持用户手动重试。

### 4.4 可扩展性

- slug 解析逻辑通过路由中间件统一处理，后续新增 slug 关联的功能（如展示主页而非聊天界面）无需大改路由结构。
- AI 助理的知识库接入通过接口抽象，支持后续替换为向量数据库、RAG 架构等方案。
- 资料字段通过独立的 `user_profiles` 表存储，与 `users` 表解耦，便于后续扩展字段而不影响核心认证逻辑。
- 完善度评分权重配置化，通过**后台管理界面**调整各字段权重，无需改动代码。

## 5. 接口预留说明（扩展点）

### 5.1 关注功能扩展点

- 个人主页 UI 预留"关注"按钮位置（MVP 阶段展示为不可点击的灰色按钮，或直接隐藏）。
- 用户资料表预留 `followers_count`、`following_count` 字段（MVP 阶段默认值为 0，不展示）。
- 后端 API 路由预留 `POST /api/v1/users/{user_id}/follow` 和 `DELETE /api/v1/users/{user_id}/follow` 端点（MVP 阶段返回 `501 Not Implemented`）。

### 5.2 私信功能扩展点

- 个人主页 UI 预留"发送消息"按钮位置（MVP 阶段隐藏或置灰）。
- 后端 API 路由预留 `POST /api/v1/messages` 端点（MVP 阶段返回 `501 Not Implemented`）。
- 数据模型层面不预先创建消息表，但在架构设计文档中规划好消息表结构，便于后续快速实现。

### 5.3 AI 助理多模型扩展点

- AI 助理调用通过 `AIProvider` 接口抽象，MVP 阶段由单一 LLM 实现，后续可接入多模型路由（如根据用户偏好或场景切换模型）。
- 知识库接入同样抽象为 `KnowledgeBaseProvider` 接口，MVP 阶段直接将 profile 文件内容注入 context，后续可替换为向量检索（RAG）方案。

**接口约定（伪代码）**：

```
interface AIProvider {
    // 发起一次对话请求
    Chat(ctx, userID, sessionID, messages []Message) → (Reply, error)
}

interface KnowledgeBaseProvider {
    // 根据用户 ID 获取知识库内容
    GetKnowledge(ctx, userID) → (string, error)

    // 更新用户的知识库内容
    UpdateKnowledge(ctx, userID, content string) → error
}
```

### 5.4 作品与动态内容扩展点

- 个人主页 UI 中预留"作品"和"动态"Tab 区域，MVP 阶段展示占位内容（"暂无内容"）。
- 后端 API 路由预留 `GET /api/v1/users/{user_id}/works` 和 `GET /api/v1/users/{user_id}/posts` 端点（MVP 阶段返回空列表）。

## 6. 数据模型概要

### 6.1 用户资料表（user_profiles）

与 `users` 表通过 `user_id` 一对一关联，独立存储扩展资料字段，避免污染核心用户表。

| 字段名              | 类型          | 约束                    | 说明                                                    |
| ------------------- | ------------- | ----------------------- | ------------------------------------------------------- |
| id                  | UUID          | PK                      | 记录唯一标识                                            |
| user_id             | UUID          | NOT NULL, UNIQUE（逻辑关联 users.id） | 关联用户，一对一                                |
| nickname            | VARCHAR(50)   | NULL                    | 昵称，未设置时为 NULL                                   |
| bio                 | VARCHAR(200)  | NULL                    | 个人简介                                                |
| avatar_url          | VARCHAR(200)  | NULL                    | 头像图片 URL（CDN 地址）                                |
| profession          | VARCHAR(50)   | NULL                    | 职业                                                    |
| skills              | TEXT[]        | NULL                    | 专长标签数组（PostgreSQL 数组类型）                    |
| interests           | TEXT[]        | NULL                    | 兴趣标签数组                                            |
| hobbies             | TEXT[]        | NULL                    | 喜好标签数组                                            |
| gender              | ENUM          | NULL                    | 枚举：male / female / undisclosed，NULL 表示未填写     |
| birthday            | DATE          | NULL                    | 生日日期                                                |
| contact_info        | VARCHAR(200)  | NULL                    | 联系方式（自由文本）                                    |
| profile_summary     | TEXT          | NULL                    | LLM 生成的 profile 摘要，NULL 表示尚未上传或生成中     |
| completion_score    | SMALLINT      | NOT NULL, DEFAULT 0     | 资料完善度分数（0-100），由后端计算并更新               |
| followers_count     | INTEGER       | NOT NULL, DEFAULT 0     | 粉丝数（预留字段，后续关注功能使用）                    |
| following_count     | INTEGER       | NOT NULL, DEFAULT 0     | 关注数（预留字段）                                      |
| created_at          | TIMESTAMP     | NOT NULL                | 创建时间                                                |
| updated_at          | TIMESTAMP     | NOT NULL                | 最后更新时间                                            |

### 6.2 用户专属短链表（user_short_links）

独立表存储用户专属短链，注册时由系统自动生成，用户不可修改。

| 字段名     | 类型          | 约束                            | 说明                                           |

| id         | UUID          | PK                              | 记录唯一标识                                   |

| short_id   | VARCHAR(16)   | NOT NULL, UNIQUE                | 系统生成的唯一标识符（字母+数字，不超过16位）  |
| created_at | TIMESTAMP     | NOT NULL                        | 创建时间                                       |

### 6.3 Profile 文件表（user_profile_files）

| 字段名           | 类型          | 约束                    | 说明                                                         |

| user_id          | UUID          | NOT NULL（逻辑关联 users.id） | 关联用户                                                     |
| file_data        | BYTEA         | NOT NULL                | 文件二进制内容（当前存储于 PostgreSQL，后续可切换存储方案）   |
