## 2. 产品概述

### 2.1 背景

BabySocial 是一款面向全行业用户的社交平台应用。用户注册与登录是平台的基础能力，所有后续功能（发布内容、社交互动、个人主页等）均依赖用户身份体系。本 PRD 定义 MVP 阶段的用户注册与登录功能范围。

### 2.2 目标

- 建立安全、可靠的用户身份体系，支撑平台后续功能迭代。
- 以邮箱为主要注册渠道，通过验证码完成邮箱真实性验证。
- 提供密码登录方式，并支持"记住我"以降低用户重复登录成本。
- 在架构层面预留手机号注册、第三方登录等扩展点，为后续版本做好准备。

### 2.3 目标用户

- **普通用户**：全行业的社交平台使用者，希望快速注册并使用平台功能。
- **管理员**：平台运营人员，负责用户管理、内容审核等后台操作。

### 2.4 MVP 范围边界

| 功能               | MVP 包含 | 后续迭代 |
| ------------------ | -------- | -------- |
| 邮箱注册           | 是       |          |
| 邮箱验证码验证     | 是       |          |
| 密码登录           | 是       |          |
| 记住我             | 是       |          |
| 人机校验（滑动图片）| 是      |          |
| 手机号注册         |          | 是       |

| 短信验证码登录     |          | 是       |
| 忘记密码 / 重置密码|          | 是       |

## 3. 功能需求

### 3.1 邮箱注册流程

#### 3.1.1 流程概述

```
填写邮箱
    ↓
人机校验（滑动图片验证）
    ↓
发送验证码至邮箱
    ↓
用户输入验证码
    ↓
设置密码
    ↓
注册完成，自动登录
```

#### 3.1.2 步骤详细说明

**步骤 1：填写邮箱**

- 用户在注册页面输入邮箱地址。
- 前端实时校验邮箱格式（RFC 5322 标准）。
- 格式不合法时，在输入框下方展示内联错误提示，禁止进入下一步。

**步骤 2：人机校验**

- 用户通过邮箱格式校验后，触发人机校验组件（MVP 版本为滑动图片验证）。
- 人机校验通过后，"发送验证码"按钮变为可点击状态。
- 人机校验组件通过抽象接口实现，支持后续替换为其他校验方式（详见第 5 节）。

**步骤 3：发送验证码**

- 用户点击"发送验证码"，系统向该邮箱发送 6 位数字验证码。
- 验证码有效期：10 分钟。
- 同一邮箱在验证码有效期内，重复请求发送验证码时，系统拒绝并提示剩余等待时间（冷却时间 60 秒）。
- 系统后端需限制同一 IP 的发送频率，防止滥用（每 IP 每小时最多 20 次）。

**步骤 4：输入验证码**

- 用户在验证码输入框中输入 6 位验证码。
- 验证码错误时，提示"验证码错误，请重新输入"，允许重试（最多 5 次，超过后需重新获取验证码）。
- 验证码过期时，提示"验证码已过期，请重新获取"，重置流程至步骤 2。

**步骤 5：设置密码**

- 用户输入密码并确认密码。
- 密码强度要求：
  - 长度 8 - 64 位
  - 至少包含字母和数字
- 两次输入不一致时，给出实时提示，禁止提交。

**步骤 6：注册完成**

- 系统创建用户账号，默认角色为"普通用户"。
- 注册成功后，系统自动为该用户颁发登录令牌，跳转至平台首页（已登录态）。

#### 3.1.3 测试阶段特殊策略（临时）

> **重要说明**：当前阶段为测试便利性考虑，同一邮箱可反复注册（即注册成功后，该邮箱可再次走完注册流程，生成新账号）。**此策略仅适用于测试阶段，正式上线前必须切换为"邮箱唯一性"校验，禁止同一邮箱重复注册。** 相关技术实现需在代码中通过配置开关控制，而非硬编码逻辑。

### 3.2 密码登录流程

#### 3.2.1 流程概述

```
输入邮箱 + 密码 + 可选"记住我"
    ↓
身份校验
    ↓
登录成功，跳转首页 / 返回令牌
```

#### 3.2.2 步骤详细说明

**步骤 1：输入凭据**

- 用户输入已注册的邮箱和密码。
- 提供"记住我"复选框，默认不勾选。

**步骤 2：身份校验**

- 后端校验邮箱是否存在、密码是否正确。
- 连续登录失败 5 次（同一邮箱），锁定该账号 15 分钟，并在页面给出提示。
- 登录失败时，统一返回"邮箱或密码错误"，不区分"邮箱不存在"和"密码错误"，防止用户枚举攻击。

**步骤 3：登录成功**

- 系统颁发 JWT 访问令牌（Access Token）和刷新令牌（Refresh Token）。
- 未勾选"记住我"：Access Token 有效期 2 小时，会话结束后失效。
- 勾选"记住我"：Refresh Token 有效期 30 天，前端自动使用 Refresh Token 续期 Access Token。
- 登录成功后跳转至平台首页。

### 3.3 用户角色

| 角色     | 说明                                               | 创建方式             |
| -------- | -------------------------------------------------- | -------------------- |
| 普通用户 | 默认角色，可使用平台社交功能                        | 注册时自动分配       |
| 管理员   | 可访问后台管理功能，具备用户管理、内容审核等权限    | 由已有管理员手动授权 |

- 用户角色信息存储于用户表，通过 RBAC 机制控制接口权限。
- MVP 阶段仅区分以上两种角色，后续可扩展更细粒度的权限控制。

## 4. 非功能需求

### 4.1 安全性

- 密码必须使用 bcrypt（cost factor >= 12）加密存储，禁止明文保存。
- 所有 API 接口必须通过 HTTPS 传输。
- JWT 密钥通过环境变量注入，禁止硬编码。
- 验证码、密码等敏感参数禁止出现在 URL Query String 中。
- 所有用户输入必须在后端进行二次校验，不依赖前端单方面校验结果。
- SQL 查询必须使用参数化查询，禁止字符串拼接，防止 SQL 注入。

### 4.2 HTTPS 证书部署方案

项目使用 JWT + HTTPS 保证传输安全，生产环境需部署 SSL/TLS 证书。推荐使用 **Let's Encrypt** 免费获取证书。

#### 4.2.1 前置条件

- 拥有一个公网可访问的域名（如 `babysocial.com`）
- 域名 DNS 已解析到服务器 IP
- 服务器开放 80 和 443 端口

#### 4.2.2 部署步骤（以 Ubuntu + Nginx 为例）

**第 1 步：安装 Certbot**

```bash
sudo apt update
sudo apt install certbot python3-certbot-nginx
```

**第 2 步：申请证书**

```bash
sudo certbot --nginx -d babysocial.com -d www.babysocial.com
```

按提示输入邮箱、同意服务条款即可。Certbot 会自动：
- 向 Let's Encrypt 验证域名所有权
- 下载证书到 `/etc/letsencrypt/live/babysocial.com/`
- 修改 Nginx 配置，启用 HTTPS 并设置 HTTP → HTTPS 自动重定向

**第 3 步：验证 HTTPS 生效**

浏览器访问 `https://babysocial.com`，确认地址栏显示锁标志。

**第 4 步：配置自动续期**

Let's Encrypt 证书有效期 90 天，Certbot 安装时已自动添加定时任务。验证自动续期是否正常：

```bash
sudo certbot renew --dry-run
```

#### 4.2.3 其他部署场景

| 场景 | 方案 |
| --- | --- |
| 使用 Docker 部署 | 在反向代理容器（如 nginx-proxy + acme-companion）中自动申请和续期 |
| 云服务平台（AWS/阿里云） | 使用平台提供的免费证书服务（如 AWS ACM、阿里云免费证书），无需手动管理 |
| 开发/测试环境 | 使用 HTTP 或自签名证书即可，不需要 Let's Encrypt |

#### 4.2.4 注意事项

- 证书私钥文件权限需限制为 `600`，仅 root 可读
- 禁止将证书私钥提交到代码仓库
- 建议配置 HSTS 头，强制浏览器使用 HTTPS

### 4.3 性能

- 注册接口（不含发送邮件）响应时间 P99 < 500ms。
- 登录接口响应时间 P99 < 300ms。
- 验证码邮件发送延迟 < 30 秒（端到端，含邮件服务商投递时间）。

### 4.4 可用性

- 注册/登录核心接口可用性 >= 99.9%。
- 邮件发送失败时，系统给出明确提示，允许用户重试，不应静默失败。

### 4.5 可扩展性

- 人机校验模块通过接口抽象，内部实现可替换（见第 5 节）。
- 认证方式（邮箱、手机号、第三方）通过策略模式设计，新增认证方式不影响现有逻辑。
- 用户角色与权限通过 RBAC 模型管理，支持后续扩展角色类型和权限粒度。

## 5. 接口预留说明（扩展点）

### 5.1 人机校验抽象接口

人机校验模块定义统一接口，MVP 阶段由"滑动图片验证"实现，后续可替换为 Google reCAPTCHA、数美、极验等方案，无需改动业务逻辑。

**接口约定（伪代码）**：

```
interface HumanVerifier {
    // 生成校验挑战（如返回图片 token）
    GenerateChallenge(ctx) → ChallengeResult

    // 验证用户完成校验的结果
    Verify(ctx, challengeID, userResponse) → (bool, error)
}
```

- 实现类通过依赖注入（DI）注入到注册服务中。
- 新增校验方式只需新增实现类，并修改 DI 配置，不修改注册流程代码。

### 5.2 手机号注册扩展点

- 用户表中预留 `phone` 字段（可为空）。
- 注册服务入口抽象为 `RegisterByEmail`、`RegisterByPhone`（待实现），共享同一套用户创建核心逻辑。
- 后端 API 路由预留 `POST /api/v1/auth/register/phone` 端点（MVP 阶段返回 `501 Not Implemented`）。

### 5.3 第三方登录扩展点

- 认证服务抽象为 `OAuthProvider` 接口，后续可对接微信、GitHub、Google 等。
- 用户表预留 `oauth_accounts` 关联表（MVP 阶段不创建，架构设计阶段规划）。
- 后端 API 路由预留 `GET /api/v1/auth/oauth/{provider}/callback` 端点（MVP 阶段返回 `501 Not Implemented`）。

## 7. 验收标准

### 7.1 邮箱注册

| 编号  | 验收场景                            | 预期结果                                                     |
| ----- | ----------------------------------- | ------------------------------------------------------------ |
| AC-01 | 输入格式不合法的邮箱并点击下一步    | 前端展示错误提示"请输入有效的邮箱地址"，无法继续             |
| AC-02 | 未完成人机校验，点击"发送验证码"   | 按钮保持不可用状态，无法触发请求                             |
| AC-03 | 完成人机校验并点击"发送验证码"     | 用户邮箱收到 6 位验证码邮件，发送按钮进入 60 秒冷却倒计时   |
| AC-04 | 60 秒冷却期内再次点击"发送验证码"  | 按钮不可用，页面显示剩余冷却时间                             |
| AC-05 | 输入正确验证码                      | 校验通过，进入密码设置步骤                                   |
| AC-06 | 输入错误验证码（累计 5 次）         | 第 5 次失败后，提示"验证码已失效，请重新获取"               |
| AC-07 | 输入过期验证码                      | 提示"验证码已过期，请重新获取"，需重新发送                   |
| AC-08 | 密码不满足强度要求                  | 页面展示密码强度要求说明，禁止提交                           |
| AC-09 | 两次密码输入不一致                  | 展示"两次密码不一致"提示，禁止提交                           |
| AC-10 | 所有步骤正确完成                    | 注册成功，用户自动登录，跳转首页，导航栏显示已登录态         |
| AC-11 | 同一邮箱重复注册（测试阶段）        | 允许注册，生成新用户账号（正式上线前此条验收标准需废止）     |

### 7.2 密码登录

| 编号  | 验收场景                            | 预期结果                                                     |
| ----- | ----------------------------------- | ------------------------------------------------------------ |
| AC-12 | 邮箱或密码错误                      | 提示"邮箱或密码错误"，不区分具体原因                         |
| AC-13 | 连续失败 5 次                       | 账号锁定 15 分钟，页面提示"账号已临时锁定，请 15 分钟后重试"|
| AC-14 | 邮箱和密码正确，未勾选"记住我"     | 登录成功，Access Token 2 小时后过期，关闭浏览器后需重新登录 |
| AC-15 | 邮箱和密码正确，勾选"记住我"       | 登录成功，30 天内重新打开浏览器无需重新登录                  |
| AC-16 | 登录成功                            | 跳转至平台首页，导航栏显示已登录态                           |

### 7.3 安全性

| 编号  | 验收场景                      | 预期结果                                           |
| ----- | ----------------------------- | -------------------------------------------------- |
| AC-17 | 查看数据库用户记录            | 密码字段为 bcrypt 加密串，无法反推原文             |
| AC-18 | 注册/登录接口通过 HTTP 访问   | 请求被拒绝或重定向至 HTTPS                         |
| AC-19 | 预留的第三方登录端点被调用    | 返回 HTTP 501 Not Implemented                      |

## 8. 业务指标

本模块作为平台基础能力，核心关注用户获取效率和认证体验质量。

### 8.1 核心指标

| 指标名称 | 定义 | 计算公式 | 目标值 | 统计口径 |
|----------|------|----------|--------|----------|
| 注册转化率 | 进入注册页面到完成注册的用户比例 | 注册完成用户数 / 注册页面访问用户数 × 100% | ≥ 60% | 日 |
| 登录成功率 | 用户提交登录表单后成功登录的比例 | 登录成功次数 / 登录提交次数 × 100% | ≥ 95% | 日 |

### 8.2 过程指标

| 指标名称 | 定义 | 计算公式 | 关注方向 | 统计口径 |
|----------|------|----------|----------|----------|
| 注册流程完成时长 | 从进入注册页到注册成功的平均时间 | SUM(注册完成时间 - 注册页进入时间) / 注册完成用户数 | 越短越好，反映流程是否顺畅 | 日 |
| 邮箱验证码验证成功率 | 用户输入验证码首次即正确的比例 | 首次验证成功次数 / 验证码提交次数 × 100% | 过低可能说明验证码邮件延迟或用户体验问题 | 日 |
| 注册漏斗各步骤流失率 | 注册流程每一步的放弃比例 | (本步骤进入人数 - 下步骤进入人数) / 本步骤进入人数 × 100% | 定位注册流程中的卡点 | 日 |
| "记住我"勾选率 | 登录时勾选"记住我"的用户比例 | 勾选"记住我"登录次数 / 总登录次数 × 100% | 反映用户信任度和使用习惯 | 周 |
| Token 刷新成功率 | Refresh Token 续期 Access Token 的成功比例 | Token 刷新成功次数 / Token 刷新请求次数 × 100% | 过低说明 Token 管理可能存在问题 | 日 |

### 8.3 健康指标

| 指标名称 | 定义 | 告警阈值 | 统计口径 |
|----------|------|----------|----------|
| 账号锁定率 | 因连续登录失败被锁定的账号占比 | > 5% 需关注，> 10% 需告警 | 日 |
| 验证码发送失败率 | 验证码邮件发送失败的比例 | > 1% 需告警 | 实时 |
| 注册接口 P99 响应时间 | 注册接口第 99 百分位响应时间 | > 500ms 需告警 | 实时 |
| 登录接口 P99 响应时间 | 登录接口第 99 百分位响应时间 | > 300ms 需告警 | 实时 |
| 人机校验通过率 | 人机校验组件的通过比例 | < 80% 需排查是否过于严格 | 日 |
